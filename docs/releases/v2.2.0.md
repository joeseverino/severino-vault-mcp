# v2.2.0 Retrieval Reliability Release

This release tightens the path from an operational question to a grounded
answer. It is aimed especially at smaller local models, where multi-step tool
use is more error-prone than the underlying retrieval.

## Highlights

- Added `get_runbook(query, limit=5)`, a single-call MCP tool that returns
  ranked hits plus the selected document body.
- Kept `secret_adjacent` content withheld in `get_runbook`; callers still need
  `read_doc(..., include_secret_adjacent=True)` plus local unlock for those
  docs.
- Moved frontmatter schema constants into `schema.py` so `server.py` and
  `doctor.py` validate against the same enum sets.
- Fixed stale write-tool validation that rejected `environment: homelab`.
- Added duplicate `doc_id` detection to `doctor`.
- Added regression coverage for the normal SSH runbook outranking SSH recovery
  procedures.

## Why It Matters

The original two-step pattern was:

```text
find_runbook(query) -> copy doc_id -> read_doc(doc_id)
```

That works well for capable clients, but it gives a local model a chance to
mistype or mutate the `doc_id`. `get_runbook` removes that failure path while
preserving the older tools for clients that prefer explicit search and read
steps.

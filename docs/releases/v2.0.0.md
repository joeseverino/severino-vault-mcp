# v2.0.0 Public Release

This is the public release of `severino-vault-mcp`: a local-first MCP server
for safely querying Git-backed Obsidian runbooks and operations inventories
with AI tools.

## Highlights

- Repositioned the project as a reusable public tool for operators,
  consultants, and security-minded teams.
- Renamed the Python package from `severino_knowledge_router` to
  `severino_vault_mcp`.
- Aligned the local checkout, installed command, Codex config, and Claude Code
  config around the public `severino-vault-mcp` name.
- Added `config.example.toml` and TOML config loading, with `SVMC_*`
  environment-variable overrides.
- Added `severino-vault-mcp doctor` to validate required frontmatter and
  `doctor --propose` to print starter frontmatter for messy vaults.
- Reworked the sample vault into a generic client-edge DNS and internal TLS
  operations scenario.
- Added `CONTRIBUTING.md` and `docs/release-checklist.md`.
- Added `docs/migration-guide.md` with a bad-doc-to-fixed-doc example and
  sensitivity triage guidance.
- Replaced the placeholder security policy with a real vulnerability-reporting
  policy using github@jseverino.com, including scope, security boundaries,
  safe configuration, and disclosure expectations.

## Security Model

- Local stdio transport only.
- No HTTP listener or hosted service.
- `secret_adjacent` bodies remain withheld by default.
- `read_doc(..., include_secret_adjacent=True)` still requires local unlock
  configuration and a successful hidden local prompt.
- Audit logs record unlock attempts without storing body content or unlock
  phrases.

## Verification

- `uv run pytest` passed with 26 tests.
- `uv run ruff check .` passed.
- Public-readiness scan found no private vault paths. Historical old-name
  references are limited to changelog and release-note migration context.
- `severino-vault-mcp doctor` validates the sample vault with no findings.

## Getting Started

- Start with `QUICKSTART.md`.
- Copy `config.example.toml` to
  `~/.config/severino-vault-mcp/config.toml`.
- Run against `examples/sample-vault/` first, then point `vault.path` at your
  own Obsidian-style operations vault.
